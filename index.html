PROYECTO 01
Kubernetes Platform

Plataforma cloud-native con app Python/Flask, 3 réplicas, HPA autoscaling, Ingress y observabilidad completa con Prometheus, Grafana y Loki.

KubernetesDockerHelmPrometheusGrafanaLoki
Completado
Qué se construyó
  • App Python/Flask containerizada con Docker multi-stage
  • Deployment con 3 réplicas y rolling updates — zero downtime
  • HPA configurado — escala de 1 a 5 réplicas según CPU
  • Ingress como único punto de entrada al cluster
  • Prometheus + Grafana + Loki — métricas, dashboards y logs centralizados
Decisiones técnicas
  • k3d para simular Kubernetes en local sin coste
  • Helm para gestionar el stack de observabilidad
  • HPA sobre métricas reales de CPU, no valores fijos
  • Ingress con nginx — un solo punto de entrada para todo el tráfico
Lo aprendido
  • Diferencia práctica entre Deployment, Service e Ingress
  • Cómo funciona el autoscaling real en Kubernetes
  • Observabilidad no es un extra — es lo que permite operar

PROYECTO 02
CI/CD Pipeline DevSecOps

Pipeline de 5 stages con GitHub Actions: validate, build Docker multi-stage, security scan con Trivy + Gitleaks, firma con Cosign y deploy automático. Todo en verde.

GitHub ActionsDockerTrivyGitleaksCosign
Completado
Qué se construyó
  • Stage 1 — Validate: lint y tests automáticos en cada push
  • Stage 2 — Build: imagen Docker multi-stage, mínima superficie de ataque
  • Stage 3 — Security: Trivy escanea CVEs + Gitleaks detecta secrets
  • Stage 4 — Sign: Cosign firma la imagen para garantizar integridad
  • Stage 5 — Deploy: actualiza el tag en Git para que ArgoCD despliegue
Decisiones técnicas
  • Seguridad integrada en el pipeline, no añadida al final
  • Multi-stage build — imagen final sin dependencias de desarrollo
  • Cosign — nadie puede modificar la imagen entre build y deploy
  • Gitleaks — secrets nunca llegan al repositorio
Lo aprendido
  • DevSecOps no es una herramienta, es una mentalidad en el pipeline
  • La firma de imágenes como garantía de supply chain security
  • Automatizar seguridad es más efectivo que revisiones manuales

PROYECTO 03
AWS Infrastructure as Code

Infraestructura AWS con Terraform modular: VPC multi-AZ, EKS cluster, RDS PostgreSQL, IAM roles y remote state en S3 con DynamoDB locking. Desarrollado con LocalStack.

TerraformAWSLocalStackEKSVPCRDS
Completado
Qué se construyó
  • VPC con subnets públicas (Load Balancer) y privadas (EKS + RDS) en 2 AZs
  • EKS cluster con endpoint privado — API server nunca expuesto a internet
  • RDS PostgreSQL con cifrado, backups automáticos y acceso restringido
  • Remote state en S3 + DynamoDB locking para trabajo en equipo seguro
  • Módulos reutilizables: vpc, eks, rds — mismo código para cualquier entorno
Decisiones técnicas
  • Terraform modular — cada módulo independiente y reutilizable
  • LocalStack para desarrollo sin coste — código 100% compatible con AWS real
  • EKS y RDS en subnets privadas — mínima exposición por diseño
  • DynamoDB locking — state nunca corrompido en entornos de equipo
Lo aprendido
  • IaC no es opcional — infraestructura manual no escala ni se puede versionar
  • El remote state es crítico en equipos — el state local es un antipatrón
  • Seguridad por diseño: redes privadas, cifrado y mínimo privilegio desde el inicio

PROYECTO 04
GitOps con ArgoCD

El proyecto que conecta todo lo anterior. Cada push desencadena el pipeline completo y ArgoCD despliega automáticamente al cluster. Rollback con un simple git revert.

ArgoCDKustomizeHelmGitHub Actionsk3d
Completado
Qué se construyó
  • Cluster k3d con ArgoCD instalado via Helm
  • Estructura Kustomize con base + overlays para dev y staging
  • ArgoCD Applications conectadas al repo de GitHub
  • Stage deploy en GitHub Actions que actualiza el tag automáticamente
  • Rollback demostrado con git revert — ArgoCD sincroniza en segundos
Decisiones técnicas
  • selfHeal: true — ArgoCD revierte cualquier cambio manual en el cluster
  • prune: true — recursos eliminados de Git desaparecen del cluster
  • Kustomize overlays — mismo base para dev y staging sin duplicar YAMLs
  • github-actions[bot] actualiza el tag — nadie toca el cluster a mano
Lo aprendido
  • GitOps no es una herramienta, es una forma de pensar
  • El cluster es un reflejo de Git, no al revés
  • El rollback más simple del mundo: git revert + git push
  • selfHeal es la killer feature — nadie puede romper producción con un kubectl accidental

PROYECTO 05
Observabilidad Avanzada con OpenTelemetry

El proyecto que cierra el ciclo. Stack completo de observabilidad con los 3 pilares: métricas, logs y trazas distribuidas correlacionadas por trace_id. Cuando algo falla, sabes dónde en minutos.

OpenTelemetryPrometheusGrafanaLokiTempoKubernetes
Completado
Qué se construyó
  • App Flask instrumentada con OpenTelemetry SDK — trazas automáticas y manuales
  • OTel Collector como hub central — desacopla la app de los backends
  • Tempo para trazas distribuidas con waterfall view en Grafana
  • Prometheus scrapeando métricas via ServiceMonitor automáticamente
  • Loki centralizando logs con correlación por trace_id
  • 4 endpoints instrumentados incluyendo simulación de errores para SLOs
Decisiones técnicas
  • OpenTelemetry vendor-neutral — cambiar de Tempo a Jaeger sin tocar el código
  • OTel Collector como hub — la app habla con un solo destino, el Collector enruta
  • Loki vs ElasticSearch — 10x más barato en recursos, indexa solo labels
  • ServiceMonitor para Prometheus — autodiscovery sin configuración manual
  • trace_id en logs — correlación directa entre trazas y logs en Grafana
Lo aprendido
  • La correlación entre señales es el verdadero valor — no las herramientas por separado
  • OpenTelemetry es el estándar — aprendes una vez, funciona en cualquier stack
  • ServiceMonitors requieren labels exactos — los detalles importan en producción
  • Los errores también se documentan — health check de Loki falla por versiones, pero las queries funcionan
